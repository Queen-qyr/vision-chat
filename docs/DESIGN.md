# AI 视觉对话助手 — 设计文档

## 1. 项目概述

本应用是一款端云协同的多模态 AI 对话产品：用户在浏览器中授权摄像头与麦克风后，可通过**语音或文字**与 AI 交流；系统在用户说话时自动截取当前画面，将「语音文本 + 画面截图」一并发送至云端多模态大模型，由 AI 结合视觉与语言上下文给出回复，并通过语音朗读或文字展示结果。

### 技术架构

```
┌─────────────────────────────────────────────────────────────┐
│                        浏览器（端侧）                          │
│  摄像头/麦克风 → 帧压缩/去重 → Web Speech API（STT）            │
│              → 对话 UI / 本地 TTS（speechSynthesis）           │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTPS（仅用户发言时携带截图）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    Node.js 后端（代理层）                       │
│  历史裁剪 / image detail=low / max_tokens 限制 / TTS 可选     │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                   OpenAI API（云端）                          │
│  gpt-4o-mini（默认）多模态对话 + tts-1（可选高质量语音）          │
└─────────────────────────────────────────────────────────────┘
```

## 3. 运营成本控制

### 3.1 想到的成本控制技巧

| 技巧 | 说明 | 预期效果 |
|------|------|----------|
| T1 | **端侧 STT**：用 Web Speech API 替代云端 Whisper | 省去全部语音识别费用 |
| T2 | **按需截图**：仅在用户完成一句话或手动触发时传图，而非视频流 | 避免按帧/按秒计费 |
| T3 | **画面压缩**：缩放到 max 512px、JPEG 0.65 质量 | 降低图片 token |
| T4 | **重复帧跳过**：连续相同画面不传图 | 减少无效 vision token |
| T5 | **image detail=low**：OpenAI 低精度图像模式 | 显著降低视觉 token |
| T6 | **历史裁剪**：仅保留最近 N 轮对话 | 控制 prompt 长度 |
| T7 | **max_tokens 限制**：回复上限 300 token | 避免冗长输出 |
| T8 | **默认 gpt-4o-mini**： cheaper 多模态模型 | 比 gpt-4o 便宜一个数量级 |
| T9 | **本地 TTS 默认**：speechSynthesis 替代 tts-1 | 省去语音合成费用 |
| T10 | **系统提示约束回复长度**：2–4 句口语化 | 间接降低 completion token |
| T11 | **场景变化才更新画面**（感知哈希/帧差） | 静态场景少传图 |
| T12 | **流式输出 + 提前截断** | 改善体验，对成本帮助有限 |
| T13 | **OpenAI Realtime API 全双工** | 体验最佳但成本最高，不采用 |

### 3.2 实际采用的技巧

| 技巧 | 实现位置 | 采用情况 |
|------|----------|----------|
| T1 端侧 STT | `client/src/hooks/useSpeechRecognition.ts` | ✅ 已采用 |

### 3.3 成本估算示例（单次对话轮次）

假设用户说一句话并附带一张低精度截图，使用 `gpt-4o-mini`：

- 文本 prompt + 截图：约 500–1500 input tokens（视历史长度而定）
- 回复：约 80–200 output tokens
- **无 STT / 无 TTS（本地）费用**
- 若使用云端 TTS：额外按字符计费（用户可选）

相比「视频流 + 云端 STT + 云端 TTS + gpt-4o + high detail」方案，单次交互成本可下降 **约 90% 以上**。

## 4. 关键设计决策

### 4.1 为何选择「发言时截图」而非持续视频流？

持续向多模态模型推送视频帧，即使每秒 1 帧，一小时也将产生 3600 次视觉推理请求，成本不可控。人类对话本身具有「轮次」结构——用户说完一句再听回复——因此在 **STT 最终结果** 时截取一帧，与语义时刻对齐，是性价比最高的方案。

### 4.2 为何 STT 放端侧、LLM 放云端？

语音识别在浏览器中已具备可用质量（Chrome/Edge 对中文支持较好），且完全免费。视觉理解与复杂推理仍依赖大模型，适合云端。该分工符合「端侧能做好的不上云」原则。

### 4.3 视觉理解准确性保障

- 使用多模态 GPT 模型，截图与用户话语同轮送入，避免图文错位
- 系统提示要求「看不清不编造」
- 用户可手动「分析当前画面」强制带图
- 可选升级 `OPENAI_MODEL=gpt-4o` 提升复杂场景识别

### 4.4 语音交互自然度

- 连续语音识别 + 临时结果显示（interim transcript）
- 回复后自动朗读
- 系统提示要求口语化、适合朗读的短句
- AI 说话时可显示状态徽章，减少用户抢话（v1 未做打断检测）
